AppForge(仮称)
GUI で作り、コードで届ける
Web アプリビルダー

Figma のように画面を描き、kintone のようにモデルを定義し、
ビルド可能な React ソースコードを生成する開発プラットフォーム構想

2026-06-12 ・ 要件定義ドラフト v0.1(requirements.md 準拠)

ビジョン Why

2つの入口、1つの成果物

  • UIファースト — Figma のようにキャンバスで画面から作る
  • モデルファースト — Salesforce / kintone のようにモデルから作る
  • どちらから始めても双方向に同期し、最終的にプロダクション品質の React コードへ

ローコードの「行き止まり」をなくす

生成物は読める・テストできる・拡張できる TypeScript。
ツールを卒業してもコードは資産として残る。

対象アプリ領域

業務 CRUD アプリ リアルタイムモニタリング 機器設定ツール Modbus データ収集・監視 ダッシュボード

ビジネスアプリに限定せず、サーバー/ネットワーク機器の監視・設定、産業プロトコル連携まで。

ゴール

  • GUI だけで画面・モデル・API を設計
  • 生成アプリは vite build がそのまま通る
  • TypeSpec を単一ソースに FE / BE / モックを生成

BE は Rust を第一候補

  • WASM モジュールとドメインロジックを共有
  • コード生成・差分計算など CPU バウンド処理のピーク性能
  • tokio-modbus 等の産業プロトコル資産
  • Go(chi/echo)への差し替えも可能なよう API 境界で疎結合に

WebAssembly の使い所

  • キャンバスのレイアウト計算(500 パーツで 60fps)
  • モデル差分検出・コード生成のホットパス

モデルデザイナー — DDD を ER 図のように Feature

キャンバスで DDD モデリング

モデルファースト → UI

モデルから CRUD 画面の雛形を自動生成。kintone 的な体験で始めて、GUI ビルダーで磨き込む。

生成コードの規約: class を使わない TypeScript

// brand pattern + companion object pattern
export type Email = string & { readonly __brand: 'Email' };

export const Email = {
  create(value: string): Result<Email, ValidationError> {
    if (!/^[^@]+@[^@]+$/.test(value))
      return err(new ValidationError('email', 'invalid'));
    return ok(value as Email);
  },
} as const;
  • brand 型でプリミティブ取り違えをコンパイルエラーに
  • データは Readonly、振る舞いは純粋関数 → テスタビリティ最大化
  • 失敗は Result<T, E> で表現(例外を投げない)
  • Vitest のユニットテスト雛形も同時生成

特定 IDL にロックインしない

  • 内部表現は IDL 非依存の中立モデル(operations / models / streams)。アダプタは import / export / diff を実装するプラグイン
  • 初期対応は TypeSpec の1種類でよいが、アダプタ I/F を最初から分離(デザイントークンと同じ構図)
  • 表現しきれない IDL 固有情報は拡張メタデータで保持しラウンドトリップを保証

コード生成は中立モデルから

  • FE クライアント: 型安全な fetch クライアント + ランタイム検証 → infra 層の repository 実装として出力(Result 型・repository I/F 準拠)
  • BE 雛形 / モックサーバーも同じモデルから生成
  • TypeSpec オーサリング時は既存 emitter(openapi3 等)をアダプタとして活用
  • 破壊的変更の差分検出は中立モデル上で IDL を問わず実行

Tailwind に依存しない、けれど連携できる

  • ドキュメントモデルには Tailwind クラス名を保存しない — スタイルは原則トークン参照で保持
  • TypeSpec と同じ「中立な中間表現 + emitter」の構図。tailwind emitter 選択時はトークン → @theme(CSS-first 設定) + ユーティリティクラスを出力
  • 逆方向も: 既存の @theme / tailwind.config からトークンへインポート

編集体験

  • プリミティブ(blue-500)→ セマンティック(primary / surface / danger)の参照構造
  • プロパティパネルはユーティリティ風のスタイル編集(4px 基準スケール = Tailwind のスケール感覚)
  • キャンバス / プレビュー / 閲覧モードには CSS 変数で即時反映 — emitter の選択と編集体験を分離
  • css-variables emitter(依存ゼロ)が常にフォールバック

生成される React アプリ Output

技術スタック

Vite 8+ React Compiler 有効 React Redux Toolkit React Router TypeScript strict Vitest

Vite 8(2026-03 安定版)はバンドラが Rolldown に統一されビルド 10〜30 倍高速 — 実行モードのビルドランナーに直接効く。生成プロジェクトの前提は Node.js 20.19+ / 22.12+。React Compiler(v1.0 安定版)が自動メモ化するため、生成コードに useMemo / useCallback は手書き出力しない。適用は reactCompilerPreset() + @rolldown/plugin-babel(plugin-react v6 の oxc 化に対応)。

品質保証

  • 生成直後に tsc --noEmit / ESLint / vite build がエラーゼロで通過
  • 再生成時、ユーザーのカスタムコード領域は保護(マーカー / ファイル分離)
  • VITE_APP_MODE=mock で全 repository が mock に切替

レイヤードアーキテクチャ + DI / DIP

src/
├ domain/          # 純粋・依存なし
│ ├ models/        # Entity / VO / Aggregate
│ └ repositories/  # I/F のみ(DIP)
├ application/     # ユースケース(I/F に依存)
├ infrastructure/
│ ├ api/           # TypeSpec 生成クライアント実装
│ └ mock/          # インメモリ mock 実装
├ presentation/    # pages / components / store
└ di/container.ts  # Composition Root で注入

domain と infra の間の repository を mock 差し替え可能に — テストもプレビューも同じ仕組みで。

ビルダーの 3 モード UX

✏️ 編集モード

GUI ビルダー / モデルデザイナー / I/F デザイナーをフル機能で操作。操作ごとの自動保存と Undo/Redo。

▶️ 実行モード

  • 生成された React アプリが実際に動作
  • 接続先: モックサーバー or 実 BE
  • BE のビルドランナーが vite build / dev server を実行しプレビュー URL を提供
  • リアルタイムパーツは WebSocket で実データ / 模擬データに接続

👀 閲覧モード(レビュー用)

  • 編集不可。だが画面遷移・ダイアログ表示は実際に操作して確認できる「動くワイヤーフレーム」
  • 画面遷移図ビュー: 全ページ + 遷移エッジ + ダイアログを俯瞰
  • パーツ・ページ・遷移・モデル要素にコメントをピン留め(スレッド / 解決済み / メンション)
  • 共有リンク(閲覧専用 URL)発行

実装方式 — DI を活かす

mock リポジトリを注入した生成 React アプリをサンドボックス(iframe + CSP)内で実行。ビルダー内インタープリタ(即時性)と mock 実行(忠実性)の併用。

コネクタ SPI — プロトコル非依存

  • コネクタは capabilities(read/write/subscribe)・設定スキーマ・コーデック・再接続ポリシーを宣言するプラグイン。設定 GUI はスキーマから自動生成
  • FE は DataChannel 抽象にのみバインド。モックコネクタへ DI 差し替え可能(閲覧モード・テスト)
  • 初期 = REST / WebSocket → M5 = Modbus → 将来 = MQTT / OPC-UA / gRPC streaming / SNMP

監視と設定の両方向

  • しきい値アラート(色変化・トースト・イベント発火)
  • 設定ツール: フォーム → 機器への書き込み(Modbus write / REST)を確認ダイアログ付きで実行
  • 通信失敗・タイムアウトの状態表示と再接続制御
  • 更新レイテンシ目標: 200ms 以内(LAN 内)
